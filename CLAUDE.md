# CLAUDE.md — AI Assistant Guide for Cardano Token Registry

## Repository Overview

This repository is the **Cardano Token Registry**, a community-governed, cryptographically-signed off-chain metadata store for Cardano native tokens. It maps on-chain asset identifiers (policyId + assetName) to human-readable metadata such as names, descriptions, tickers, and logos.

- **API endpoint**: `https://tokens.cardano.org/metadata` (read-only RESTful API)
- **Scope**: Cardano native tokens only (no ADA)
- **Governance**: All changes go through GitHub Pull Requests with automated + human review

## Token Metadata File Format

Each file in `mappings/` is a JSON object. Below is the full schema:

```json
{
    "subject": "<base16-encoded policyId + assetName>",
    "policy": "<base16-encoded CBOR monetary policy script>",
    "name": {
        "sequenceNumber": 0,
        "value": "Human Readable Token Name",
        "signatures": [
            {
                "signature": "<hex-encoded Ed25519 signature>",
                "publicKey": "<hex-encoded public key>"
            }
        ]
    },
    "description": {
        "sequenceNumber": 0,
        "value": "Human-readable description of the token.",
        "signatures": [
            {
                "signature": "<hex-encoded Ed25519 signature>",
                "publicKey": "<hex-encoded public key>"
            }
        ]
    },
    "ticker": {
        "sequenceNumber": 0,
        "value": "TKR",
        "signatures": [...]
    },
    "url": {
        "sequenceNumber": 0,
        "value": "https://example.com",
        "signatures": [...]
    },
    "logo": {
        "sequenceNumber": 0,
        "value": "<base64-encoded PNG image>",
        "signatures": [...]
    },
    "decimals": {
        "sequenceNumber": 0,
        "value": 6,
        "signatures": [...]
    }
}
```

### Required Fields
| Field | Description |
|---|---|
| `subject` | Base16-encoded concatenation of `policyId` + `assetName` |
| `name` | Human-readable token name (signed object) |
| `description` | Human-readable description (signed object) |

### Optional Fields
| Field | Description |
|---|---|
| `policy` | Base16-encoded CBOR monetary policy script |
| `ticker` | Short ticker symbol (signed object) |
| `url` | HTTPS web page URL (signed object) |
| `logo` | PNG image as base64 byte string (signed object) |
| `decimals` | Integer decimal places for display (signed object) |

### Signed Object Pattern

All mutable metadata fields follow this structure:
```json
{
    "sequenceNumber": 0,
    "value": "<field value>",
    "signatures": [
        {
            "signature": "<hex Ed25519 signature>",
            "publicKey": "<hex public key>"
        }
    ]
}
```

The `sequenceNumber` increments on updates. Signatures are produced with `token-metadata-creator` and provide cryptographic proof that the submitter controls the minting policy.

## Creating a New Token Entry

Token metadata entries are created with the `token-metadata-creator` tool (available inside `nix-shell`):

```bash
# Initialize a new entry
token-metadata-creator entry --init <subject>

# Add required fields
token-metadata-creator entry <subject> \
  --name "My Token" \
  --description "A description of my token" \
  --policy path/to/policy.json

# Add optional fields
token-metadata-creator entry <subject> \
  --ticker "MTK" \
  --url "https://mytoken.io" \
  --logo path/to/logo.png \
  --decimals 6

# Sign the entry
token-metadata-creator entry <subject> -a path/to/signing.skey

# Finalize
token-metadata-creator entry <subject> --finalize
```

This produces `mappings/<subject>.json`.

## Submission Rules

When creating or reviewing PRs that modify token metadata, enforce these rules:

1. **Single commit** directly off the `master` branch
2. **Single file** added or modified per PR
3. **File location**: Only files inside `mappings/` are permitted — no other paths
4. **Filename** must exactly match the `subject` field value (lowercase hex)
5. **Max file size**: 370KB per entry
6. **Subject format**: Lowercase base16-encoded `policyId + assetName`

Violations of these rules cause CI to fail and the PR to be rejected.

## CI/CD Pipeline

The GitHub Actions workflow at `.github/workflows/ci.yaml` runs automatically on every pull request.

### Job: `Validate-Metadata`

**Trigger**: Pull requests to `master`
**Runner**: `ubuntu-latest`

**Steps**:
1. Checkout the PR branch and `master` branch side-by-side
2. Validate that only files inside `mappings/` were changed
3. Download the latest validation binaries from IOHK Hydra:
   - `metadata-validator-github`
   - `token-metadata-creator`
4. For each changed file:
   - **Added (A)**: Validates the PR version only
   - **Modified (M)**: Validates both the master version and PR version

**Validation commands**:
```bash
# GitHub PR context validation
./metadata-validator-github <owner> <repo> <PR#> \
  --expect-branch master --no-auth

# Schema and signature validation
token-metadata-creator validate <file1> [<file2> ...]
```

### What CI Checks

- File is located in `mappings/` directory
- File size ≤ 370KB
- JSON is well-formed
- Required fields (`subject`, `name`, `description`) are present
- `subject` matches the filename
- Cryptographic signatures are valid
- Field values conform to format constraints (ticker length, URL scheme, etc.)

## Common Tasks for AI Assistants

### Validating a Metadata File
```bash
# Inside nix-shell
token-metadata-creator validate mappings/<subject>.json
```

### Checking File Size
```bash
wc -c mappings/<subject>.json
# Must be < 370,000 bytes (370KB)
```

### Verifying Subject Matches Filename
```bash
# Extract subject from file and compare to filename
jq -r '.subject' mappings/<subject>.json
```

### Listing All Registered Subjects
```bash
ls mappings/ | sed 's/\.json$//'
```

### Searching for a Token by Name
```bash
jq -r '.name.value' mappings/*.json | grep -i "TokenName"
```

### Checking for Duplicate Tickers
```bash
jq -r '.ticker.value // empty' mappings/*.json | sort | uniq -d
```

## Key Constraints and Gotchas

- **No traditional tests**: Validation is entirely handled by CI tools (`token-metadata-creator`, `metadata-validator-github`). There is no test suite to run locally beyond the Nix-provided toolchain.
- **No scripting languages**: The repo contains no Python, JavaScript, or Ruby. All tooling is provided via Nix as compiled binaries.
- **Signatures are mandatory**: Every mutable field must have valid cryptographic signatures. Files cannot be manually edited without re-signing.
- **Subject is immutable**: The `subject` field and filename cannot change after registration.
- **`sequenceNumber` must increment**: When updating an existing entry, all modified fields must have a higher `sequenceNumber` than the current master version.
- **Logo format**: Must be a PNG image encoded as a base64 string. Max size counts toward the 370KB file limit.
- **URL must use HTTPS**: The `url` field must begin with `https://`.
- **Ticker length**: Typically 2–9 characters (validated by `token-metadata-creator`).
